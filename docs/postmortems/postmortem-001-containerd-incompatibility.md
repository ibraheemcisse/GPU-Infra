# Postmortem 001: containerd 2.x breaks kubeadm on Kubernetes 1.31

**Date:** 2026-05-31  
**Severity:** Critical — cluster initialization blocked entirely  
**Duration:** ~8 hours across two sessions  
**Resolution:** Downgrade containerd to 1.7.29, regenerate config  
**Status:** Resolved

---

## Summary

A Kubernetes 1.31 cluster built on AWS EC2 (Ubuntu 22.04) failed to initialize because the bootstrap script installed containerd 2.2.4 from Docker's apt repository. containerd 2.x introduced breaking changes to both the CRI implementation and the config file format. kubeadm could not communicate with the container runtime, the control plane never came up, and several hours were spent diagnosing symptoms rather than the root cause.

---

## Timeline

**Session 1**

- Bootstrap script runs on control plane. Packages install cleanly including containerd 2.2.4.
- `kubeadm init` fails with: `unknown service runtime.v1.RuntimeService`
- containerd config generated by the bootstrap script was version 3 format, incompatible with containerd 1.7.x assumptions
- Assumed config issue. Regenerated config.toml, restarted containerd. kubeadm init succeeds on manual retry.
- Worker node bootstrapped. `kubeadm join` succeeds.
- Cilium installed. Worker Cilium pod enters CrashLoopBackOff.
- Diagnosis shifts to inter-node networking. Hours spent on SG rules, NACLs, iptables, route tables.
- tcpdump shows packets leaving worker but never arriving at control plane.
- Instances terminated and redeployed twice. Problem persists.
- Session ends. Root cause unknown.

**Session 2**

- Fresh control plane. containerd 2.2.4 installed again by bootstrap script.
- Operator in second chat identifies the version mismatch immediately.
- containerd downgraded to 1.7.29. Config regenerated. kubeadm init succeeds cleanly.
- Cilium installs and reaches OK status on first attempt.
- Worker joins. Both nodes Ready.
- Root cause confirmed.

---

## Root Cause

The bootstrap script installs containerd with:

```bash
apt-get install -y containerd.io
```

Docker's apt repository serves the latest version. As of May 2026, that is containerd 2.2.4. containerd 2.x introduced two breaking changes relevant to this setup:

**1. Config file format version 3**

containerd 2.x generates a version 3 config by default. containerd 1.7.x cannot parse version 3 config and exits with:

```
containerd: failed to load TOML from /etc/containerd/config.toml: unsupported config version `3`
```

The bootstrap script runs `containerd config default` immediately after installing the package. On containerd 2.x, this generates a v3 config. If containerd is later downgraded to 1.7.x for any reason, or if the config is read by a 1.7.x-compatible tool, it fails.

**2. CRI v1 runtime API changes**

containerd 2.x changed parts of the CRI implementation. kubeadm 1.31 connecting to containerd 2.2.4 via `/run/containerd/containerd.sock` receives:

```
rpc error: code = Unimplemented desc = unknown service runtime.v1.RuntimeService
```

This is not a socket issue or a config issue. It is a version incompatibility between the Kubernetes 1.31 CRI client expectations and the containerd 2.x server implementation.

## Symptoms and Misdiagnoses

Because the control plane never fully initialized, the symptoms looked like other problems.

**Symptom:** Cilium on worker CrashLoopBackOff, `no route to host` to API server  
**Misdiagnosis:** Inter-node networking blocked by AWS, SG rules, NACL, or routing  
**Reality:** The control plane was never stable. Cilium could not connect to the API server because the API server had issues stemming from the broken runtime layer.

**Symptom:** `kubectl get nodes` refused connection  
**Misdiagnosis:** kubeconfig not set, kube-apiserver not running  
**Reality:** Correct diagnosis but wrong root cause identified. The API server was not running because containerd was broken.

**Symptom:** `unknown service runtime.v1.RuntimeService`  
**Misdiagnosis:** containerd socket path wrong, CRI disabled, config.toml missing  
**Reality:** containerd version incompatibility with Kubernetes 1.31.

**Symptom:** Packets not arriving between nodes (tcpdump)  
**Misdiagnosis:** AWS hypervisor bug, VPC networking issue  
**Reality:** Cilium had partially initialized and corrupted host routing. The broken state was caused by containerd instability, not AWS.

## Resolution

**Immediate fix (applied manually):**

```bash
# Downgrade containerd
sudo apt-get install -y --allow-downgrades \
  containerd.io=1.7.29-1~ubuntu.22.04~jammy

# Delete broken config
sudo rm -f /etc/containerd/config.toml

# Regenerate with 1.7.x defaults
containerd config default | sudo tee /etc/containerd/config.toml

# Enable systemd cgroup driver
sudo sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' \
  /etc/containerd/config.toml

# Restart
sudo systemctl restart containerd
```

**Permanent fix (applied to bootstrap scripts):**

Both `bootstrap-control-plane.sh` and `bootstrap-gpu-worker.sh` updated to pin the containerd version:

```bash
# Before
apt-get install -y containerd.io conntrack

# After
apt-get install -y --allow-downgrades \
  containerd.io=1.7.29-1~ubuntu.22.04~jammy conntrack
```

Committed: `fix: pin containerd to 1.7.29 for Kubernetes 1.31 compatibility`

## What Made This Hard to Diagnose

The error `unknown service runtime.v1.RuntimeService` is not a well-known failure mode in most kubeadm guides. Most documentation assumes containerd 1.x. containerd 2.x was released in 2024 and the incompatibility is not prominently documented.

The failure was also non-obvious because the bootstrap script appeared to succeed. Packages installed, containerd started, the service showed active. The problem only surfaced when kubeadm tried to use the runtime.

The secondary symptoms (routing issues, CrashLoopBackOff, packet loss) were all real but were consequences of the primary failure, not independent problems. Debugging the secondary symptoms first added hours of investigation time.

---

## Detection

This issue would have been caught earlier by adding a CRI validation step to the bootstrap script:

```bash
# Verify CRI is functional before proceeding
crictl --runtime-endpoint unix:///run/containerd/containerd.sock info \
  | grep -q '"status": true' || { echo "CRI not ready"; exit 1; }
```

A failing CRI check immediately after containerd install would have surfaced the root cause in minutes rather than hours.
